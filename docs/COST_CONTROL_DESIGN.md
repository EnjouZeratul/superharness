# SuperHarness 成本控制功能设计规范

> 版本: v3.0 (简化版)
> 日期: 2026-05-09
> 基于: 八轮专家评审 + 用户真实需求验证

---

## 一、设计原则修正

### 1.1 用户真实需求（第八轮评审结论）

**用户质疑**："企业不需要超预算自动停，只需要显示用量和预估"

**专家评审结论**：用户质疑有**60-70%合理性**

| 需求层次 | 用户真正想要的 | 竞品方案 |
|---------|---------------|---------|
| **P0 核心** | 看到花了多少钱 | OpenAI Dashboard, LangSmith |
| **P1 重要** | 知道下一步要花多少 | 大多数竞品缺失 |
| **P2 可选** | 预算超限提醒 | OpenAI Usage Alerts |
| **P3 特殊** | 防止失控循环 | max_turns, timeout |

**关键洞察**：
> 用户要的是"手电筒"（看清问题），不是"刹车"（强行停止）

### 1.2 竞品分析结论

| 竞品 | 有无"自动停" | 成本控制方式 |
|------|-------------|-------------|
| OpenAI API | ❌ 无 | 速率限制 + 告警 + 月度统计 |
| LangSmith | ❌ 无 | 可观测性 + 成本分析 |
| AutoGen | ❌ 无 | max_turns + timeout |
| CrewAI | ❌ 无 | Human oversight |

**结论**：竞品不做"自动停"是因为：
1. 破坏任务完整性
2. 中途停止浪费已调用部分
3. 更好的方案是监控+告警

### 1.3 简化后的设计原则

```
错误设计：复杂的阈值系统 + 暂停逻辑 + Checkpoint恢复
正确设计：用量可见 + 预估透明 + 非侵入式预警 + 可选防护

核心转变：
- 从"控制用户"变为"赋能用户"
- 从"复杂配置"变为"开箱即用"
- 从"核心功能"变为"观测工具"
```

## 二、MVP功能集（简化版）

### 2.1 核心功能（必须有）

```
MVP成本控制 = {
    1. 实时用量显示
       ├── 当前已用 tokens
       ├── 当前对话成本（货币）
       └── 累计用量统计

    2. 下一步预测
       ├── 预计 tokens 消耗（基于历史统计）
       └── 预计成本

    3. 非侵入式预警
       ├── 达到预算80%时 UI 提示
       └── 达到预算100%时 日志警告
}
```

### 2.2 延后功能（Phase 2+）

```
延后功能 = {
    1. 月度预算硬限制（管理员配置）
    2. 详细用量报表
    3. 项目/用户级预算拆分
    4. 异常模式检测
}
```

### 2.3 删除的功能

| 功能 | 删除原因 |
|------|---------|
| 5级阈值系统（50/70/90/100/150%） | 过度设计，用户配置负担重 |
| 超预算暂停+用户选择 | 侵入性过强，破坏任务完整性 |
| Checkpoint保存恢复机制 | 这是灾难恢复，不是预算管理 |
| 复杂的BudgetConfig API | 配置复杂，用户需要阅读文档 |
| 原子操作保护 | 过度工程化 |

## 三、API设计（简化版）

### 3.1 用量追踪（核心API）

```python
from superharness import UsageTracker

# 自动追踪用量，无需配置
tracker = UsageTracker()

# 执行时自动记录
response = await agent.run("分析PDF")

# 查看用量
print(tracker.current_usage)
# UsageReport(
#     prompt_tokens=1500,
#     completion_tokens=500,
#     total_tokens=2000,
#     cost_usd=0.03,
#     model="gpt-4"
# )

# 预测下一步消耗
estimate = tracker.estimate_next_step()
# TokenEstimate(
#     min_tokens=200,
#     max_tokens=800,
#     likely_tokens=400,
#     cost_range="$0.02-$0.08"
# )
```

### 3.2 预算提醒（可选配置）

```python
# 简单的预算提醒，只有1个阈值
from superharness import BudgetAlert

budget = BudgetAlert(
    monthly_limit=100.0,      # 月度预算
    warning_at=0.8,          # 80%时警告
    notify=log_warning        # 非侵入式通知
)

# 绑定到Agent
agent.set_budget_alert(budget)

# 超过80%时：
# 控制台输出: ⚠️ 预算使用已达 80% ($80.00/$100.00)
# 不暂停，不弹窗，继续执行
```

### 3.3 失控防护（特殊场景）

```python
# 可选：防止失控循环（仅开发/测试环境）
from superharness import SafetyLimit

safety = SafetyLimit(
    max_tokens_per_task=100000,  # 单任务上限
    max_iterations=50             # 最大迭代次数
)

agent.set_safety_limit(safety)

# 仅在极端情况触发：
# "任务已达到安全限制 (100k tokens)，可能存在失控循环"
```

## 五、对比修正

### 5.1 设计对比

| 维度 | v2.0（复杂版） | v3.0（简化版） |
|------|---------------|---------------|
| 阈值级别 | 5级（50/70/90/100/150%） | 1级（80%预警） |
| 用户配置 | BudgetConfig（10+参数） | BudgetAlert（2参数） |
| 超预算行为 | 暂停+弹窗+选择 | 非侵入式警告 |
| Checkpoint | 自动保存恢复 | 仅失控时保存 |
| 核心定位 | 成本控制功能 | 可观测性工具 |

### 5.2 实现复杂度对比

| 维度 | v2.0 | v3.0 |
|------|------|------|
| 开发时间 | 4-5周 | 1-2周 |
| API数量 | 15+ | 5 |
| 配置项 | 10+ | 2-3 |
| 用户学习成本 | 高（需读文档） | 低（开箱即用） |

## 六、其他创新点评审结论

### 6.1 自嗨检测结果

| 创新点 | 自嗨指数 | 结论 |
|--------|---------|------|
| Token预算可视化 | 65/100 | ✅ 保留核心，简化配置 |
| Agent执行录像回放 | 85/100 | ⚠️ 延后Phase 2+，用结构化日志替代 |
| Agent健康诊断 | 70/100 | ⚠️ 简化为超时+循环检测 |
| 合规审计日志 | 25/100 | ✅ 保留，企业版功能 |
| 手动Checkpoint | 55/100 | ⚠️ 简化为手动checkpoint |

### 6.2 修正后的创新点优先级

```
MVP阶段（必须有）：
├── 实时用量追踪 + 下一步预测
├── 基础错误重试（3次指数退避）
└── 结构化日志输出

Phase 1（重要但延后）：
├── 本地Dashboard（Web界面）
├── 手动Checkpoint API
└── 超时检测 + 循环检测

Phase 2（锦上添花）：
├── 录像回放（如有真实需求）
├── 健康诊断（简化版）
└── 月度预算硬限制

企业版：
├── 合规审计日志
├── 项目级预算管理
└── 用量报表和导出
```

## 七、总结

### 7.1 核心认知修正

```
第八轮评审前：
"成本控制 = 复杂的阈值系统 + 暂停逻辑 + Checkpoint恢复"

第八轮评审后：
"成本控制 = 用量可见 + 预估透明 + 非侵入式预警 + 可选防护"

关键转变：
- 删除"超预算自动停"的复杂逻辑
- 保留"显示用量和预估"的核心价值
- 将"自动停"降级为可选防护（防止失控循环）
```

### 7.2 成功关键

```
好的预算功能是让用户忘记它的存在，
直到真正需要它的时候。

当前设计让用户时刻意识到预算限制，
这是错误的。
```

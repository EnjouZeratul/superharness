# SuperHarness 项目愿景：如何做出真正"Super"的项目

> 版本: v1.0
> 日期: 2026-05-09
> 基于: 六位专家深度分析（技术/产品/AI研究/市场/工程/开发者体验）

---

## 执行摘要

**核心问题**：如何让 SuperHarness 不是"小打小闹"，而是真正有吸引力、有实用性的项目？

**核心答案**：不是做更多功能，而是做一件事做到 95 分——让 Agent 系统可调试、可预测、可恢复。

---

## 一、用户真正的痛点（不是我们想象的）

### 1.1 用户最头疼的问题（按优先级）

| 优先级 | 痛点 | 具体场景 | 框架能解决吗 |
|--------|------|----------|-------------|
| **P0** | **不可预测性** | "不知道 Agent 下一步会做什么" | ✅ 主要责任 |
| **P0** | **成本失控** | "一觉醒来发现账单 $200" | ✅ 主要责任 |
| **P0** | **调试困难** | "出了问题不知道哪里错" | ✅ 主要责任 |
| **P1** | **可靠性差** | "同样的输入，结果波动大" | 🔶 框架可改善 |
| **P1** | **集成困难** | "框架与现有系统对接成本高" | ✅ 可解决 |
| **P2** | **测试困难** | "怎么确保 Agent 对所有输入都正确" | ✅ 可解决 |

### 1.2 企业 vs 个人开发者需求差异

| 需求 | 企业用户 | 个人开发者 |
|------|---------|-----------|
| 成本控制 | **极度敏感**（预算审批） | 中等敏感 |
| 稳定性要求 | **99%+ 成功率** | 70-80% 可接受 |
| 调试能力 | 需要团队可理解 | 自己能看懂 |
| 合规要求 | 数据隐私、审计日志是硬性要求 | 通常不关心 |
| 文档质量 | **关键因素**（影响团队采用） | 次要 |

### 1.3 真实场景中的用户痛点

**场景一：Agent 进入无限循环**
> "Agent 不断重复调用同一个工具，消耗了 $50 API 费用后我发现它仍在循环"

**场景二：Agent "忘记"之前的指令**
> "对话进行到第 15 轮，Agent 开始重复之前已经完成的工作"

**场景三：多 Agent 协调失败**
> "Agent A 说已完成，Agent B 说没收到，我找不到是哪里出了问题"

**场景四：无法复现线上问题**
> "线上 Agent 失败了，但我拿到日志后根本看不懂它当时在想什么"

## 二、竞品真正解决不了的问题

### 2.1 竞品能力空白

| 问题 | LangChain | SmolAgents | CrewAI | 空白程度 |
|------|-----------|------------|--------|----------|
| **成本硬性控制** | 需 LangSmith（付费） | 无（HF 主推免费模型） | 无 | **极高** |
| **确定性保证** | 无 | 无 | 无 | **极高** |
| **本地零配置调试** | 需 LangSmith（付费） | 无 | 无 | **极高** |
| **问题诊断** | 无 | 无 | 无 | **极高** |
| **从断点恢复** | 需配置 | 无 | 无 | **高** |

### 2.2 为什么竞品不会做

**LangChain**：
- 商业利益冲突：LangSmith 是付费产品
- 定位是"完整生态"，不会做本地免费版

**SmolAgents**：
- 极简定位冲突：成本控制、错误恢复需要复杂机制
- HF 主推免费模型，用户无成本焦虑

**CrewAI**：
- 定位是多 Agent 协作，不专注执行引擎

## 三、什么是真正的"Super"

### 3.1 不是功能多，而是解决核心问题

**错误思路**：
- ❌ "我们有 15 个创新点"
- ❌ "LangChain 有的我们也有"
- ❌ "比对手功能更多"

**正确思路**：
- ✅ "解决用户最头疼的一个问题"
- ✅ "竞品不愿意做或做不到的事"
- ✅ "一件事做到 95 分"

### 3.2 "Super" 级架构的标准

| 维度 | 现有框架 | "Super" 标准 |
|------|---------|-------------|
| 可预测性 | 同样输入，结果波动 | 确定性模式可选 |
| 成本控制 | 事后统计 | 事前预估 + 硬性上限 |
| 调试能力 | 日志淹没关键信息 | 诊断 + 问题定位 |
| 稳定性 | 一错全崩 | 断点恢复 + 优雅降级 |
| 透明度 | 黑盒执行 | 每步可观测可追溯 |

### 3.3 三大核心价值主张

**价值一："执行时告诉你花了多少，下一步要花多少"**
```python
# 其他框架
response = await agent.run("任务")  # 不知道花了多少钱

# SuperHarness - 简单易用
from superharness import UsageTracker

tracker = UsageTracker()
response = await agent.run("任务")

# 实时查看用量
print(tracker.current_usage)
# UsageReport(tokens=2000, cost_usd=0.03)

# 预测下一步
estimate = tracker.estimate_next_step()
# TokenEstimate(min=500, max=1500, cost_range="$0.05-$0.15")
```

**设计原则（第八轮评审修正）**：
- 用户要的是"手电筒"（看清问题），不是"刹车"（强行停止）
- 从"控制用户"变为"赋能用户"
- 从"复杂配置"变为"开箱即用"
- 删除过度设计：5级阈值、暂停逻辑、Checkpoint机制（预算场景）

**价值二："Agent 不应该因为一个错误就崩溃"**
```python
# 其他框架
try:
    response = await agent.run("爬取100个网站")
except Exception:
    pass  # 失败了就失败了

# SuperHarness
try:
    response = await agent.run("爬取100个网站")
except AgentError:
    # 用户主动恢复（如果需要）
    await agent.resume_from_checkpoint()
```

**价值三："零配置就能调试"**
```bash
$ superharness dashboard
# 打开 localhost:8080，零配置看到所有执行细节
```

## 四、具体可执行的技术方向

### 4.1 核心：可靠的 Agent 执行引擎

**架构设计**：
```
Agent Execution Engine
├── Execution Graph (任务图)
│   ├── Node Types: Task, Condition, Loop, Parallel
│   ├── Edge Types: Dependency, Data Flow
│   └── State Management: Input, Output, Context
├── Runtime
│   ├── Scheduler (执行调度)
│   ├── Observer (状态追踪)
│   └── Recovery (失败恢复)
└── Debugger
    ├── Step-by-step Execution
    ├── Breakpoints & Watch
    ├── Time-travel Debugging (回放历史状态)
    └── Cost & Performance Metrics
```

**工作量估算**：
- MVP（线性执行 + 状态追踪）：1-2 个月
- 完整版（并行 + 条件 + 回放）：2-3 个月

### 4.2 技术突破点

| 方向 | 难度 | 价值 | 建议 |
|------|------|------|------|

| **成本实时计数** | 低 | 高 | **优先实现** |
| **错误模式检测** | 中 | 高 | **优先实现** |
| **诊断助手** | 高 | 高 | Phase 2 |
| **规划可靠性** | 高 | 高 | Phase 3 |

### 4.3 前沿技术的落地可行性

**差的错误消息**：
```

```

**好的错误消息**：
```
❌ Agent execution failed

📍 Location: WeatherAgent.execute() in weather_agent.py:42

🔍 Root cause: Tool 'get_weather' returned unexpected format

Expected: {"temperature": number, "condition": string}
Received: {"temp": 25, "condition": "sunny"}

💡 Suggested fixes:
   1. Check tool 'get_weather' output schema
   2. Add output parser in tool definition
   3. See docs: /docs/tools/output-parsing
```

## 六、差异化定位

### 6.1 不做什么

| 不做 | 原因 |
|------|------|
| 比对手更轻量 | SmolAgents 已占领这个心智 |
| 比对手功能更多 | LangChain 已占领，且不是竞争力 |
| 追求"零依赖" | 用户不关心依赖数量 |
| 复杂的规划策略 | 整合型创新，易被复制 |

### 6.2 要做什么

| 要做 | 原因 |
|------|------|
| **成本硬性控制** | 竞品不关注，用户刚需 |
| **确定性保证模式** | 竞品没有，解决"不敢用"的核心障碍 |
| **本地零配置调试** | LangSmith 付费，这是空白地带 |
| **问题诊断** | 没有任何框架做到 |

### 6.3 一句话定位

**SuperHarness 是简洁可靠的 Agent 运行时。**

## 八、成功关键

### 8.1 做 1 个 95 分的功能

**公式**：
```
成功 = (首次体验成功 × 核心问题解决) / 学习成本
```

**核心原则**：
- 70 分的功能在竞争中被淹没
- 95 分的功能会形成口碑传播
- 1 个 95 分功能的维护成本是可控的

### 8.2 三大差异化支柱

**支柱一：可控能力**
- 确定性控制点
- 行为约束系统
- 审批流程集成

**支柱二：透明的成本**
- 硬性预算控制
- 实时消耗追踪
- 成本优化指导

**支柱三：可调试的黑盒**
- 问题模式自动识别
- 原因分析与建议
- 时间旅行调试

## 十、总结

### 核心洞察

1. **用户不在乎功能多少**，在乎"能不能快速解决问题"
2. **竞品的技术质量被广泛批评**，但它解决的是"信任问题"
3. **空白地带存在**：成本控制、本地调试、确定性保证
4. **小团队的机会**：做一件事做到 95 分，而不是十件事做到 70 分

### 最终建议

> **做一个 95 分的 Agent 执行引擎，让复杂的多步骤任务可调试、可回溯、可恢复。**
>
> 不是更多的工具，不是更酷的演示，而是让 Agent 系统变得可靠和可预测。

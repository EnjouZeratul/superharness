## 三、调度机制

### 进度触发流程

```
开发完成 → 组长初审 → 审查员复审 → 副组长终审 → 助理更新看板
    ↓          ↓          ↓            ↓
  (不通过)   (不通过)   (不通过)      (通过)
    ↓          ↓          ↓            ↓
  返工修复   返工修复   返工修复     通知相关方
```

### 阻塞升级机制

| 阻塞时长 | 升级路径 |
|----------|----------|
| < 30分钟 | 组长解决 |
| 30分钟 - 1小时 | 副组长协调 |
| > 1小时 | Team Lead决策 |

### 每日汇报机制

```
助理每日汇总 → 更新看板 → 同步文档 → 通知Team Lead
```

## 四、审查标准

### DEMO/MVP检测清单

审查员必须验证以下项目：

#### 代码层面
- [ ] 无 `todo!()` / `unimplemented!()` / `pass`
- [ ] 无硬编码返回值
- [ ] 无模拟数据替换真实实现
- [ ] 完整错误处理（所有Result/Exception有处理）
- [ ] 类型注解完整（Python）

#### 功能层面
- [ ] LLM调用真实API（抓包/日志验证）
- [ ] 数据持久化有效（重启验证）
- [ ] 错误提示用户友好
- [ ] 配置化完成（非硬编码）

#### 测试层面
- [ ] 单元测试覆盖核心逻辑
- [ ] 边界条件测试
- [ ] 错误路径测试

### 审查结果

| 结果 | 动作 |
|------|------|
| ✅ 通过 | 更新看板，通知下一依赖方 |
| ⚠️ 需修改 | 记录问题，开发者修复后重新审查 |
| ❌ 不通过 | 返工重写，副组长介入 |
